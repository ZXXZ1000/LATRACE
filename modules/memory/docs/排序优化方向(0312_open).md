# 排序优化方向（0312 open）

## 背景

本轮分析基于 `benchmark/suites/showcase/outputs/latency_split_validation_v5/showcase_debug.md` 的 19 道题结果，以及 `modules/memory` 当前线上实现与配置。

当前 showcase 的结果是 `15/19 = 78.9%`，但这个数字低估了排序问题的严重程度。原因是：有些题虽然最终答对了，但正确证据并不在前排，而是被 QA 模型从更靠后的 evidence 里“救回来”。

这说明当前问题不只是“有 4 道题答错”，而是：

- 候选召回不够精确
- 候选排序对 answer-bearing evidence 不够友好
- 配置层和 `dialog_v2` 的真实行为并不一致

## 现象

### 1. 基础事实题会被同一批无关事件反复击中

`q01` 和 `q02` 是最典型的例子：

- `q01`: “Mia 正在做的内部工具项目代号叫什么？”  
  预期答案：`Aurora`
- `q02`: “Mia 把供应商联系人主要记在哪里？”  
  预期答案：`Notion`

但两题的前几条 evidence 几乎一样，都是：

- 停瑜伽改骑车
- 靠过道
- 早上喝什么

见：

- `benchmark/suites/showcase/outputs/latency_split_validation_v5/showcase_debug.md:23-40`
- `benchmark/suites/showcase/outputs/latency_split_validation_v5/showcase_debug.md:42-59`

这说明当前 retrieval 在 query selectivity 上明显不足。不同问题没有把候选池有效拉开。

### 2. 成功题也暴露出“排序弱于答案”

`q03` 与 `q04` 最值得注意：

- `q03` 最终答对了“茉莉花茶”，但前几条 evidence 仍然主要是“靠过道 / 骑车”。  
  见 `benchmark/suites/showcase/outputs/latency_split_validation_v5/showcase_debug.md:61-87`
- `q04` 最终答对了“苏州”，但前几条 evidence 仍然是“骑车 / 靠过道 / 咖啡”。  
  见 `benchmark/suites/showcase/outputs/latency_split_validation_v5/showcase_debug.md:89-115`

这说明：

- QA 最终能答对，不代表 retrieval rank 健康
- 当前 top-15 里可能有正确证据，但排序位置偏后
- 排序质量比最终 accuracy 更差

### 3. 更新题容易召回到 topic，却召不回 answer slot

`q05` 问的是：

- “Aurora 的后端后来是从什么框架改成了什么框架？”
- 预期答案：`从 Flask 改成 FastAPI`

但 top evidence 基本都是：

- Aurora pilot
- Aurora Assist
- 苏州出差

而没有任何 `Flask / FastAPI`。  
见 `benchmark/suites/showcase/outputs/latency_split_validation_v5/showcase_debug.md:117-139`

这说明当前系统更容易找对“话题”，而不是找对“回答问题所需的具体槽位”。

## 当前检索链路的真实行为

### 1. `dialog_v2` 不是单层排序，而是两层排序叠加

第一层是 `store.search()` 内部的局部排序：

- 向量召回
- BM25 rerank
- graph/recency 加权

见：

- `modules/memory/application/service.py:1015-1142`
- `modules/memory/application/service.py:1298-1376`

第二层是 `dialog_v2` 自己的 route fusion：

- `E_event_vec`
- `E_vec`
- `K_vec`
- `EN`
- `T`
- 再做 RRF + recency + graph signal + multi-route bonus

见：

- `modules/memory/retrieval.py:1380-2018`

这意味着：当前系统其实有两个排序器，但只有第一层部分由 YAML 控制。

### 2. `memory.config.yaml` 不是 `dialog_v2` 的唯一真相

`memory.config.yaml` 里确实有：

- `memory.search.rerank.alpha_vector/beta_bm25/gamma_graph/delta_recency`
- `memory.search.ann.default_topk`
- `memory.api.topk_defaults.retrieval`

见：

- `modules/memory/config/memory.config.yaml:173-230`

但 `dialog_v2` 的关键参数其实硬编码在 `retrieval.py`：

- `_DIALOG_V2_DEFAULT_WEIGHTS`
- `candidate_k=50`
- `seed_topn=15`
- `graph_cap=5`
- `rrf_k=90`

见：

- `modules/memory/retrieval.py:182-190`
- `modules/memory/retrieval.py:1383-1395`

同时，HTTP `/retrieval` 并没有暴露这些参数：

- `RetrievalDialogBody` 只允许 `topk / strategy / backend / tkg_explain / entity_hints / time_hints`

见：

- `modules/memory/api/server.py:2397-2412`
- `modules/memory/api/server.py:2760-2780`

结论：

- 改 `memory.config.yaml` 可以影响底层 `store.search()` 的局部排序
- 但改不了 `dialog_v2` 的主融合逻辑

## 高置信原因

### 一、当前并没有真正的 lexical first-stage，BM25 只是 vector hits 上的二次排序

`MemoryService.search()` 的执行顺序是：

1. `vectors.search_vectors(...)`
2. 对这些 vector hits 建 BM25 corpus
3. BM25 只在这批 hits 上重排

见：

- `modules/memory/application/service.py:1015-1039`
- `modules/memory/application/service.py:1087-1142`

这意味着：

- `Aurora / Notion / Flask / FastAPI` 如果没进向量候选，BM25 无法补召回
- 当前不是“语义 + 词面”并联召回，而是“语义主导，词面补排”

这也是为什么 showcase 这种只有几十条对话、但答案词面非常明确的场景，仍然会丢 `Notion / Flask / FastAPI`。

### 二、`E_event_vec` 对 topic 友好，但对 answer slot 不友好

`dialog_v2` 的事件主路由是：

- 先搜 `tkg_dialog_event_index_v1`
- 用 `summary + desc` 的 event 文本做向量检索

见：

- `modules/memory/retrieval.py:1580-1624`
- `modules/memory/application/graph_service.py:329-379`

而 event index 的文本来源只是：

- `Event.summary`
- `Event.desc`

这会天然有两个问题：

1. 如果抽取器在 `summary/desc` 里没保留关键槽位，后面永远搜不到
2. event 文本更像“主题摘要”，不一定保留所有精确事实

所以：

- `q05` 能抓到很多 “Aurora 相关事件”
- 但抓不到 “Flask -> FastAPI” 这种细粒度技术槽位

### 三、`graph_cap=5` 会先硬保留 event route，进一步放大 topic 偏置

`dialog_v2` 的 candidate selection 不是统一排序后取前 K，而是：

1. 先拿 `E_event_vec` 的前 `graph_cap=5`
2. 再用 `_base_score` 补齐剩余候选

见：

- `modules/memory/retrieval.py:1924-1957`

这会导致：

- 只要 `E_event_vec` 前几名偏题，前面的位置就先被占掉
- 其他 route 即使更适合 answer-bearing evidence，也只能在后半段补位

这里需要实话实说：

- 我没有做正式 A/B
- 所以不能直接断言“去掉 `graph_cap` 一定全局更好”

但就这次 showcase 的失败样式看，`graph_cap` 很大概率在伤害 `q01/q02/q05` 这种短事实题和更新细节题。

### 四、`run_id` 看起来传了，但默认 scoping 让它经常不真正生效

`E_vec` 和 `K_vec` 在 route 构造时都带了 `run_id`：

- `modules/memory/retrieval.py:1632-1640`
- `modules/memory/retrieval.py:1720-1728`

Qdrant 也支持 `metadata.run_id` 的精确过滤：

- `modules/memory/infra/qdrant_store.py:215-218`

但 `MemoryService.search()` 的默认 scoping 是：

- `default_scope = domain`
- `fallback_order = [session, domain, user]`

见：

- `modules/memory/config/memory.config.yaml:183-187`
- `modules/memory/config/runtime_overrides.json`

更关键的是，实际尝试顺序是：

- 先按 `domain`
- 再按 `session`
- 再按 `user`

见：

- `modules/memory/application/service.py:893-917`

这意味着：

- 即便 caller 传了 `run_id`
- 只要 domain 级已经有 hits
- 就不会真正进入 session 级检索

所以当前系统的真实行为更接近：

- 统一记忆池优先
- session 只是 fallback 尝试之一

这本身不是绝对错误，但会带来一个后果：

- 旧事实、重复事实、后续相似事实更容易混进同一个候选池
- 在短对话闭集场景里，precision 会明显下降

### 五、事件构图阶段缺少语义去重，重复事件会浪费 rank budget

当前 `dialog_tkg_graph_v1` 会对 `events_raw` 逐条建 Event：

- 没有看到真正的语义 dedup
- 也没有对 paraphrase 事件做 collapse

见：

- `modules/memory/domain/dialog_tkg_graph_v1.py:400-490`

而写入前的 policy 也相对宽松：

- `min_event_confidence = 0.4`
- `max_events_per_session = 0`

见：

- `modules/memory/config/memory.config.yaml:119-128`
- `modules/memory/application/config.py:226-256`
- `modules/memory/session_write.py:227-258`

这和 debug 里的现象一致：

- `q01/q02` 前两条是高度相似的“停瑜伽改骑车”
- `q05` 里同一个 `Aurora pilot` 事件也出现了多个近重复版本

这些重复 event 会直接消耗 top-k。

### 六、`entity_route` 对中文对话中的英文人名过于“宽召回”

`entity_route` 默认会从 query 里抓：

- `face123`
- 英文 Title Case 名字

见：

- `modules/memory/retrieval.py:1763-1772`

在 showcase 里，很多问题都写着 `Mia ...`。  
这意味着 entity route 很可能总是在抓 `Mia`，然后通过 `graph_list_events()` 拿出大量 Mia 相关事件：

- `modules/memory/retrieval.py:1807-1827`

这会带来一个低精度 broad recall：

- 几乎所有 Mia 的事件都可能入池
- 再和 `event_vec` / `multi-route bonus` 叠加
- 泛相关事件会越来越强

我不能仅凭当前 debug 报告证明它一定是主因，因为对外 debug 没有把 route counts 全展开。  
但从实现上看，这是一个高概率噪声源。

### 七、`time_route` 对中文月度时序表达几乎无能为力

当前 `_parse_time_window()` 能识别的主要是：

- `YYYY-MM-DD`
- 今天 / 昨天
- 上周 / 上个月

见：

- `modules/memory/retrieval.py:194-240`

它并不支持：

- `三月`
- `二月开始`
- `后来`
- `最终`

这意味着 showcase 里很多 temporal/update 题，其实不是 time route 在工作，而是别的 route 在“语义上碰对了”。

### 八、`signal` 算了但不用，排序没吃到 explain 的收益

`dialog_v2` 会跑 `tkg_explain_event_evidence` 并算 `_graph_signal`：

- `modules/memory/retrieval.py:1960-1990`

但默认权重是：

- `signal = 0.0`

见：

- `modules/memory/retrieval.py:182-190`

所以这块现在的效果更像：

- 增加了成本
- 帮 QA 展示 evidence
- 但没有帮助 final rerank

### 九、模型选择不是唯一根因，但它在放大上述问题

当前配置：

- embedding: `openai/text-embedding-3-small`
- extractor: `google/gemini-2.5-flash`
- qa: `qwen-plus`

见：

- `modules/memory/config/memory.config.yaml:10-23`
- `modules/memory/config/memory.config.yaml:93-109`

我的判断是：

- QA 不是主瓶颈，失败主要来自 evidence 不对
- extractor 和 embedding 更可能是瓶颈放大器

尤其你们的所有 route 本质上都还是 embedding-first，这会把 embedding 质量放大成系统瓶颈。

## 这件事最核心的判断

当前系统更擅长：

- 找到“跟这个人/这个主题相关”的事件

还不够擅长：

- 找到“能回答这个问题的那一句 / 那个精确槽位”

换句话说：

- 它更像 topic/event retrieval engine
- 还不是一个对 exact slot QA 友好的 memory ranker

这就是为什么：

- `Aurora / Notion / Flask / FastAPI` 这种本来应该很简单的题
- 在 60 条对话里仍然会失败

## 优先级建议

### P0

- 给 `dialog_v2` 增加真正的 lexical first-stage，不能只靠 BM25 rerank vector hits
- 把 `dialog_v2` 的关键参数从硬编码里抽出来，至少能受配置或 API 控制
- 重新梳理 scoping 语义，明确 `run_id` 在 `dialog_v2` 里到底是不是有效约束

### P1

- 重新评估 `E_event_vec` 的权重和 `graph_cap`
- 做 event-level semantic dedup，避免重复事件吃掉 top-k
- 让 `time_route` 支持中文月度与相对时态表达
- 让 `graph_signal` 进入真正的排序实验，而不是只做装饰性 explain

### P2

- 重新评估 embedding 模型是否适合中文短对话 + 英文专有词混合场景
- 评估 extractor 是否在 event summary/desc 中稳定保留 answer-bearing 槽位

## 最后一句结论

这不是“某个权重轻微调一下”的问题。  
当前 showcase 暴露出的，是 `memory` 在 **索引建模、route 设计、作用域语义、融合排序** 四层上的系统性偏 topic、弱 lexical、弱 slot-selectivity。

如果不先把这个判断立住，后面只靠 `topk`、`weights`、`更强模型` 去补，效果会很有限。

## 固定 State 的四道错题拆解（2026-03-13）

本节基于同一份固定 state 做的复盘：

- state: `benchmark/suites/showcase/outputs/ab_baseline_local_20260312/showcase_state.json`
- baseline A/B: `benchmark/suites/showcase/outputs/hybrid_reuse_ab_baseline_state_20260313/results.json`
- baseline debug: `benchmark/suites/showcase/outputs/ab_baseline_local_20260312/showcase_debug.md`

核心结论先说：

- 这 4 道题对应的正确信息 **都已经写进了系统**
- 原始 utterance、UtteranceEvidence、Event、Knowledge 多数都存在
- 失败主因不是“没 ingest / 没抽取到”，而是 **正确证据排序太靠后**

### q08 · 生日礼物最终选择

- 原始对话中存在明确事实：
  - `Mia: 我最后送了她一支钢笔，还附了一张手写明信片。`
- Qdrant 中存在：
  - 原始 utterance index
  - 初始计划事实 `正在考虑送钢笔`
- Neo4j 中存在：
  - `Event.summary = Mia送了Lin一支钢笔和一张手写明信片作为生日礼物`

但在实际 retrieval 中：

- top-15 没有返回最终礼物事件
- 把旧计划 `想送钢笔` 排在了更前面
- 当 `topk=50` 时，正确事件只出现在 **rank 26**

结论：

- 主因：**排序问题**
- 次因：更新态 evidence 没有被优先于旧计划 evidence

### q11 · 运动习惯更新

- 原始对话中存在明确事实：
  - `二月开始就停了，我现在改成周日早上沿河骑车。`
- Qdrant 中存在：
  - 原始 utterance：旧习惯 `每周二晚上做瑜伽`
  - 原始 utterance：新习惯 `周日早上骑车`
  - fact：`Mia在二月开始停止了每周二的瑜伽`
- Neo4j 中存在两个关键 Event：
  - `Mia停止了每周二的瑜伽`
  - `Mia现在改成周日早上沿河骑车`

但在实际 retrieval 中：

- top-15 里能看到旧瑜伽和“还在坚持周日骑车”的弱相关证据
- 真正表达“从瑜伽改成骑车”的 Event 只在 `topk=50` 时出现在 **rank 23 / 24**

结论：

- 主因：**排序问题**
- 细化判断：不是完全没召回，而是“决定性 replacement evidence 不在前排”
- 这题说明当前排序更容易保留 topic 相近证据，不够偏向 answer-bearing update evidence

### q12 · demo 时间

- 原始对话中存在明确事实：
  - `演示定在 5 月 22 日下午，我这周都在为那天做准备。`
- Qdrant 中存在：
  - 原始 utterance
  - fact：`Aurora 的演示日期是 2025 年 5 月 22 日下午`
- Neo4j 中存在：
  - `Event.summary = Aurora 的演示定在 5 月 22 日下午`

但在实际 retrieval 中：

- top-15 主要还是 Aurora 相关的“框架改造 / 展位物料 / pilot”事件
- 正确 demo 时间事件只在 `topk=50` 时出现在 **rank 28**
- knowledge fact 甚至更靠后，在 **rank 35**

结论：

- 主因：**排序问题**
- 这题非常典型地说明：系统能找到 `Aurora` 这个 topic，但找不到回答 `when` 所需的时间槽位

### q15 · one-on-one 改期

- 原始对话中存在明确事实：
  - `因为要准备 demo，那场 one-on-one 改到周五上午了。`
- Qdrant 中存在：
  - 原始 utterance
- Neo4j 中存在：
  - `Event.summary = one-on-one 会议改到周五上午`
  - `desc = 因为要准备 demo，原定周四下午的 one-on-one 会议改期。`

但在实际 retrieval 中：

- top-15 没有把这条 meeting-update 事件排进来
- 当 `topk=50` 时，正确事件只出现在 **rank 27**

结论：

- 主因：**排序问题**
- 这题说明当前系统对“会议安排更新”这种 work-scheduling slot 的优先级不足

## 这一轮复盘的总判断

固定 state 下，`q08/q11/q12/q15` 的共同点非常清楚：

- 不是数据污染
- 不是没有 tenant/user/session 隔离
- 不是抽取器完全漏掉了关键信息
- 而是 **正确证据在库里，但前排 rank budget 被其他 topic-similar evidence 吃掉**

因此，下一阶段优化重点不应再放在：

- 单纯换一组 rerank 权重
- 或单纯把 lexical_hybrid 打开

更应该优先处理：

1. update / temporal / meeting 这类 answer-bearing evidence 的前排优先级
2. 旧事实 vs 新事实的显式更新排序偏置
3. 事件摘要与 knowledge fact 对时间槽位、变更槽位的 selective boost
